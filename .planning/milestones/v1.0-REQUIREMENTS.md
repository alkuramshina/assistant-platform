# Requirements

## Phase 4

- [x] **REFRAME-01**: Docs identify SSH-deployed multi-bot console as the active direction.
- [x] **ARCH-01**: Architecture covers Deployer, console, bot deployment engine, persistence, logs, and isolation.
- [x] **ARCH-02**: Coolify and hosted control planes are out of scope for prototype.

## Phase 5

- [x] **INST-01**: Deployer connects to a Linux server over SSH.
- [x] **INST-02**: Deployer verifies OS, Docker, Compose, disk, memory, and network prerequisites.
- [x] **INST-03**: Deployer requests approval before installing missing prerequisites.
- [x] **INST-04**: Deployer deploys/updates console under `/opt/nanobot-console`.

## Phase 6

- [x] **CONS-01**: Console persists bot records.
- [x] **CONS-02**: Console stores secret references, not raw secrets in logs.
- [x] **CONS-03**: Console API supports create/list/start/stop/status/logs.

## Phase 7

- [x] **BOT-01**: Console renders per-bot Compose files.
- [x] **BOT-02**: Each bot has isolated data, workspace, secrets, and Compose project name.
- [x] **BOT-03**: Bot containers do not mount `docker.sock`.
- [x] **BOT-04**: Telegram allowlist is required before deploy.

## Phase 8

- [x] **UI-01**: UI creates a bot.
- [x] **UI-02**: UI lists bots and statuses.
- [x] **UI-03**: UI starts/stops bots and shows logs.

## Phase 9

- [x] **SMOKE-00**: Deployer prepares the VM/server, starts the console, and prints the UI URL; no manual `mkdir`, `scp`, or `python -m console` is required for the happy path.
- [x] **SMOKE-01**: Allowlisted Telegram user receives a response from a deployed bot.
- [x] **LOG-01**: Activity log records user request and assistant response with bot ID and timestamp.
- [x] **LOG-02**: Activity log records provider/model/status/error.

## Phase 10

- [x] **ISO-01**: Two bots run on one server without shared state.
- [x] **ISO-02**: Stopping one bot does not stop the other.
- [x] **HARD-01**: Runtime safety constraints are verified.
